# 仓库内容公开建议

更新日期：2026-04-08

## 目的

这份清单用于帮助在首次公开仓库前，快速判断哪些文件适合继续公开保留，哪些更适合迁移到 `docs/`，哪些建议不要直接进入开源仓库。

## 建议总览

### 建议公开保留

| 文件 | 建议 | 原因 |
|---|---|---|
| `README.md` | 保留在根目录 | 公开仓库首页入口 |
| `CONTRIBUTING.md` | 保留在根目录 | 协作者上手入口 |
| `SECURITY.md` | 保留在根目录 | GitHub 安全页可直接识别 |
| `CODE_OF_CONDUCT.md` | 保留在根目录 | 公开协作行为约定 |
| `SUPPORT.md` | 保留在根目录 | 对外支持与反馈入口 |
| `ROADMAP.md` | 保留在根目录 | 公开路线图，适合替代内部开发计划 |
| `docs/project-structure.md` | 保留 | 维护者与贡献者的结构入口 |
| `docs/open-source-checklist.md` | 保留 | 公开前准备清单 |
| `docs/first-publish-guide.md` | 保留 | 首次推送 GitHub 的操作指引 |
| `交付部署说明.md` | 可公开保留或迁移到 `docs/` | 如果项目主打自部署和离线交付，这份文档对外有价值 |
| `火山引擎参考.md` | 建议保留，但可迁移到 `docs/integrations/` | 已去除真实 Key，适合作为第三方接入参考 |

### 建议迁移到 `docs/` 后再公开

| 文件 | 建议 | 原因 |
|---|---|---|
| `关键词大全.md` | 迁移到 `docs/creative-reference.md` 或类似路径 | 内容本身可公开，但当前命名更像内部素材库 |
| `原始需求设想.md` | 迁移到 `docs/history/` 或 `docs/vision.md` | 适合作为项目愿景背景，不适合直接作为根目录主文档 |

### 建议默认不公开或至少先不放根目录

| 文件 | 建议 | 原因 |
|---|---|---|
| `工作交接文档.md` | 默认不公开 | 面向内部接手，带有明显的项目交接语境 |
| `当前阶段工作总结.md` | 默认不公开 | 属于阶段性内部总结，公开价值低，且可能持续暴露开发状态细节 |
| `开发计划.md` | 视情况保留，建议转为公开 Roadmap 后再发布 | 当前更像内部执行基线，而非面对社区的公开路线图 |
| `正式验收清单.md` | 默认不公开 | 偏交付验收与内部流程，不是开源协作优先信息 |
| `节点预设增强执行计划.md` | 默认不公开 | 属于专项执行计划，公开后维护成本高 |
| `AGENTS.md` | 视情况决定 | 如果你想公开 AI 协作方式可保留，否则更适合作为本地协作约定 |

## 建议动作

1. 根目录优先只保留“对外入口文档”。
2. 仍有公开价值但不适合放根目录的文件，统一迁移到 `docs/`。
3. 明显面向内部交接、验收、专项执行的文件，首次公开时建议先不纳入仓库。
4. 如果你希望公开项目演进过程，可把 `开发计划.md`、`原始需求设想.md` 改写成更面向社区的 `ROADMAP.md` 或 `docs/history/` 文档。

## 一句话建议

首次公开时，根目录越克制越好。优先留下 `README`、贡献说明、安全说明、协作准则和少量高价值技术文档，其余内容按“公开价值 > 维护成本 > 信息边界风险”的顺序逐步放出。
