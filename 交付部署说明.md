## 2. 范围与边界

当前项目支持三种交付方式：

- 源机器本地在线 Docker 运行
- 源机器生成离线交付包后，目标机器离线部署运行
- 源机器生成绿色包后，目标机器免安装启动运行

这里的“离线交付”只表示：

- 目标机器部署时不需要重新拉取 Docker 镜像
- 目标机器部署时不需要重新安装 npm / pip 依赖

这里的“绿色包”表示：

- 目标机器不需要预装 Docker
- 目标机器不需要预装 Python / npm
- 源机器打包时已经把前端产物、Python 运行时和依赖整合进可分发目录

它**不等同于**“目标机器完全不需要外网”。

如果目标机器要继续使用火山引擎主链路，需要满足：

- 已通过前端“工具栏 -> 引擎”保存可用的 Ark 配置，或在部署前已在 `.env` / `.env.docker` 中预置默认值
- 目标机器能够访问 `ARK_BASE_URL`

如果目标机器完全隔离外网：

- Docker 部署本身仍可完成
- 绿色包启动本身也仍可完成
- 但火山引擎生成功能不可用
- 此时只能使用 `mock` 或后续接入的其他可访问适配器

另外需要说明：

- 当前产品正式名已经统一为 `SketchShot`
- 但仓库目录名、镜像名、环境变量前缀、部分兼容脚本仍保留 `WXHB` 命名，用于兼容已有交付链路

## 3. 相关文件说明

## 4. 环境变量映射关系

源机器在线 Docker 运行使用 `.env.docker`，变量名前缀为 `WXHB_`。

目标机器离线包内运行使用 `.env`，变量名不带 `WXHB_` 前缀。

离线打包脚本会自动做映射，常见字段关系如下：

| 源机器 `.env.docker` | 目标机器 `.env` |
|---|---|
| `WXHB_BACKEND_PORT` | `BACKEND_PORT` |
| `WXHB_FRONTEND_PORT` | `WEB_PORT` |
| `WXHB_DEFAULT_ADAPTER` | `DEFAULT_ADAPTER` |
| `WXHB_VOLCENGINE_ENABLED` | `VOLCENGINE_ENABLED` |
| `WXHB_ARK_API_KEY` | `ARK_API_KEY` |
| `WXHB_PUBLIC_BASE_URL` | `PUBLIC_BASE_URL` |

说明：

- 源机器如果已正确填写 `.env.docker`，打包后生成的离线包 `.env` 会自动继承这些值
- 如果某些值未配置，离线包会保留 `.env.offline.example` 中的默认值
- 如果 `WXHB_APP_NAME` 仍是历史值，打包脚本会自动收敛为 `SketchShot - AI Storyboard Canvas`
- 这些火山变量现在主要用于首次启动默认值、无人值守部署或无法进入前端时的兜底配置
- 启动后，业务用户更推荐直接在前端“工具栏 -> 引擎”中维护 Ark 地址、API Key 和模型 ID
- 前端保存后的配置会写入 `backend\data\engine_config.json`（在线运行）或离线包 `data\engine_config.json`（离线部署）

## 5. 前置条件

### 5.1 源机器打包前

- 当前项目代码完整
- 如果要生成离线 Docker 包，已安装并启动 Docker Desktop
- 如果要生成绿色包，已安装 Python 3 与 npm / Node.js，且当前项目前后端依赖可用
- 如需首次启动就预置火山配置，可在 `.env.docker` 中填写 `WXHB_ARK_API_KEY` 与模型参数
- 若构建依赖下载较慢，可在 `.env.docker` 中配置 `WXHB_PIP_INDEX_URL` 与 `WXHB_PIP_TRUSTED_HOST`

### 5.2 目标机器部署前

- 已拿到打包生成的离线目录或压缩包
- 如需真实调用火山引擎，目标机器网络可访问 `ARK_BASE_URL`
- 如果要部署离线 Docker 包，已安装并启动 Docker Desktop 或 Docker Engine，且能正常执行 `docker` 与 `docker compose`
- 如果要运行绿色包，不要求 Docker，但建议确认 `8000` 端口未被其他服务占用

## 6. 推荐交付流程

标准交付流程建议按下面顺序执行：

1. 源机器在线自检
2. 选择生成离线交付包或绿色包
3. 交付给目标机器
4. 目标机器部署或直接启动
5. 目标机器校验
6. 按正式验收清单做人工验收

### 6.1 源机器在线自检

建议先在源机器确认 Docker 在线运行正常：

```powershell
.\scripts\docker-build.ps1 -Action up
.\scripts\docker-build.ps1 -Action ps
```

默认访问地址：

- 前端：`http://localhost:8080/`
- 后端健康检查：`http://localhost:8000/api/health`

### 6.2 源机器离线打包

在项目根目录执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Package-OfflineBundle.ps1
```

也可以直接双击：

```text
scripts\Build-OfflineBundle.cmd
```

执行后会自动完成：

- 构建前后端 Docker 镜像
- 将镜像保存为离线 tar 包
- 复制离线部署所需的 compose、env、部署脚本、校验脚本、清理脚本
- 生成目录 `offline-bundle\时间戳\`
- 同时生成一个可直接分发的压缩包 `offline-bundle\时间戳.zip`

如果镜像已经构建好，只想重新生成离线包，可以执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Package-OfflineBundle.ps1 -SkipBuild
```

如果希望强制重新构建镜像，可执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Package-OfflineBundle.ps1 -NoCache
```

### 6.3 源机器绿色包打包

在项目根目录执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Package-GreenBundle.ps1
```

也可以直接双击：

```text
scripts\Build-GreenBundle.cmd
```

如果前端已经提前构建好，只想基于现有前端产物出包，可执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Package-GreenBundle.ps1 -SkipFrontendBuild
```

如果需要显式指定源机器上的 Python，可执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Package-GreenBundle.ps1 -PythonExe "C:\Python\Python311\python.exe"
```

执行后会自动完成：

- 复用或构建 `frontend/dist`
- 整合后端代码与默认独立模式配置
- 整合本机 Python 运行时与当前环境所需依赖
- 收敛交付根目录，只保留 `Run-SketchShot.cmd`、`Stop-SketchShot.cmd`、`README.txt`
- 将内部脚本、日志目录和清单文件收进 `_internal\`
- 生成目录 `green-bundle\时间戳\`
- 同时生成一个可直接分发的压缩包 `green-bundle\时间戳.zip`

### 6.4 交付给目标机器

将以下任意一种交付给目标机器：

- `offline-bundle\时间戳\` 整个目录
- `offline-bundle\时间戳.zip` 压缩包
- `green-bundle\时间戳\` 整个目录
- `green-bundle\时间戳.zip` 压缩包

推荐直接交付 zip。

如果交付的是离线 Docker 包，由目标机器解压后部署。

如果交付的是绿色包，由目标机器解压后直接双击启动。

### 6.5 目标机器部署

Windows 目标机器推荐直接双击：

```text
Start-SketchShot.cmd
```

等价命令为：

```powershell
powershell -ExecutionPolicy Bypass -File .\Deploy-OfflineBundle.ps1
```

执行后会自动完成：

- 导入离线镜像 `wxhb-images-offline.tar`
- 使用 `docker-compose.offline.yml` 启动前后端容器
- 自动执行健康检查
- 输出访问地址

默认访问地址通常为：

- 前端：`http://localhost:8080/`
- 后端健康检查：`http://localhost:8000/api/health`

如果需要调整端口或火山引擎配置，可在部署前先修改离线包里的 `.env`。

补充建议：

- 如果是交付给最终使用方，推荐先完成部署，再引导用户从前端右上角“工具栏 -> 引擎”中填写自己的 Ark 地址、API Key 和模型 ID
- 前端保存后的配置会落到离线包 `data\engine_config.json`，后续重启容器仍会保留

### 6.6 绿色包目标机器启动

绿色包目标机器推荐直接双击：

```text
Run-SketchShot.cmd
```

绿色包特点：

- 目标机器不需要 Docker
- 目标机器不需要 Python / npm
- 包内会优先使用 `runtime\python\python.exe`
- 默认日志写入 `_internal\.runtime\`
- 默认运行数据写入 `backend\data-standalone\`

如果需要停止绿色包，可双击：

```text
Stop-SketchShot.cmd
```

如果需要排障，可查看：

- `_internal\bundle-manifest.json`
- `_internal\.runtime\`
- `backend\.env.standalone`（首次启动后生成）

### 6.7 离线 Docker 目标机器部署后校验

Windows 目标机器可双击：

```text
Verify-SketchShot.cmd
```

等价命令为：

```powershell
powershell -ExecutionPolicy Bypass -File .\Verify-OfflineBundle.ps1
```

该脚本会校验：

- backend 容器是否 `running` 且 `healthy`
- frontend 容器是否 `running` 且 `healthy`
- 后端健康接口是否返回正常
- 前端页面是否可以正常访问

### 6.8 离线 Docker 目标机器清理

Windows 目标机器可双击：

```text
Cleanup-SketchShot.cmd
```

等价命令为：

```powershell
powershell -ExecutionPolicy Bypass -File .\Cleanup-OfflineBundle.ps1
```

该操作会：

- 停止并删除离线部署容器
- 删除离线镜像标签
- 默认保留 `data` 目录中的运行数据

如果需要彻底清理运行数据，可执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\Cleanup-OfflineBundle.ps1 -PurgeData
```

## 7. scripts 目录文件作用说明

### 7.1 源机器脚本

- `scripts\docker-build.ps1`
  源机器在线 Docker 运行管理脚本。支持 `build`、`up`、`down`、`restart`、`logs`、`ps`、`config`。

- `scripts\Package-OfflineBundle.ps1`
  离线交付包打包主脚本。负责构建镜像、离线镜像打包、生成离线目录、复制部署脚本、生成 `.env`、生成 zip 压缩包。

- `scripts\Build-OfflineBundle.cmd`
  Windows 双击版离线打包入口。内部调用 `Package-OfflineBundle.ps1`。

- `scripts\Package-GreenBundle.ps1`
  绿色包打包主脚本。负责整合前端产物、后端代码、Python 运行时、依赖、极简入口脚本并输出 zip 压缩包。

- `scripts\Build-GreenBundle.cmd`
  Windows 双击版绿色包打包入口。内部调用 `Package-GreenBundle.ps1`。

### 7.2 目标机器脚本

- `scripts\Deploy-OfflineBundle.ps1`
  离线包部署主脚本。负责加载镜像、启动容器、调用校验脚本、输出访问地址。

- `scripts\Verify-OfflineBundle.ps1`
  离线部署校验脚本。负责检查容器状态、健康检查结果与前端访问结果。

- `scripts\Cleanup-OfflineBundle.ps1`
  离线部署清理脚本。负责停止容器、删除离线镜像标签，并可选清理数据目录。

- `scripts\Start-SketchShot.cmd`
  Windows 双击版部署入口。内部调用 `Deploy-OfflineBundle.ps1`。

- `scripts\Verify-SketchShot.cmd`
  Windows 双击版校验入口。内部调用 `Verify-OfflineBundle.ps1`。

- `scripts\Cleanup-SketchShot.cmd`
  Windows 双击版清理入口。内部调用 `Cleanup-OfflineBundle.ps1`。

- `scripts\Start-WXHB.cmd`、`scripts\Verify-WXHB.cmd`、`scripts\Cleanup-WXHB.cmd`
  历史兼容入口，保留用于兼容旧交付习惯。

- `scripts\Deploy-OfflineBundle.sh`、`scripts\Verify-OfflineBundle.sh`、`scripts\Cleanup-OfflineBundle.sh`
  Linux 目标机器对应脚本。

## 9. 交付包目录结构说明

### 9.1 离线包目录结构说明

离线打包完成后，生成目录大致如下：

```text
offline-bundle\20260404_xxxxxx\
  .env
  .env.example
  docker-compose.offline.yml
  wxhb-images-offline.tar
  wxhb-images-offline.sha256.txt
  Deploy-OfflineBundle.ps1
  Verify-OfflineBundle.ps1
  Cleanup-OfflineBundle.ps1
  Start-SketchShot.cmd
  Verify-SketchShot.cmd
  Cleanup-SketchShot.cmd
  Start-WXHB.cmd
  Verify-WXHB.cmd
  Cleanup-WXHB.cmd
  Deploy-OfflineBundle.sh
  Verify-OfflineBundle.sh
  Cleanup-OfflineBundle.sh
  README_OFFLINE.txt
  data\
    engine_config.json  （运行后按需生成）
```

说明：

- `.env` 是可直接运行的目标机配置
- `.env.example` 是参考模板
- `wxhb-images-offline.tar` 是前后端镜像离线包
- `data\` 是目标机器上的本地持久化目录
- `data\engine_config.json` 会在用户首次通过前端保存引擎配置后自动生成

### 9.2 绿色包目录结构说明

绿色包打包完成后，生成目录大致如下：

```text
green-bundle\20260410_xxxxxx\
  README.txt
  Run-SketchShot.cmd
  Stop-SketchShot.cmd
  _internal\
    Start-SketchShot.ps1
    Stop-SketchShot.ps1
    Verify-SketchShot.ps1
    bundle-manifest.json
  backend\
    app\
    data-standalone\
    .env.standalone.example
  frontend\
    dist\
  runtime\
    python\
```

说明：

- 根目录只暴露最终用户真正需要的启动 / 停止入口
- `_internal\` 用于存放内部脚本、日志目录和清单文件
- 首次启动后会在 `backend\` 下生成 `.env.standalone`
- 默认运行数据会写入 `backend\data-standalone\`
- 绿色包启动不依赖 Docker

## 10. 交付前建议重点确认

正式交付前，建议至少确认以下事项：

- `.env.docker` 中端口配置是否符合预期
- 如需真实火山引擎，确认已采用以下两种方式之一：
  源机器 / 目标机器的环境变量中预置了默认值
  或实际验收机器已在前端“工具栏 -> 引擎”中保存了可用配置
- 如需真实火山引擎，确认源机器与目标机器都能访问 `ARK_BASE_URL`
- 源机器已成功生成 zip 包
- 如果走绿色包交付，确认源机器打包环境中的 Python 依赖完整可用
- 目标机器 Docker 已正常运行
- 目标机器未被其他服务占用 `8000` 或 `8080` 端口
- 目标机器已按 [正式验收清单](./正式验收清单.md) 预留验收步骤

补充说明：

- 如果走绿色包交付，目标机器不要求 Docker，但仍建议确认 `8000` 端口未被其他服务占用
- 如果走绿色包交付，建议在一台干净机器上先双击验证 `Run-SketchShot.cmd`

## 11. 一句话操作指南

如果只记一套最短流程，可以按下面理解：

1. 源机器执行  
   `powershell -ExecutionPolicy Bypass -File .\scripts\Package-OfflineBundle.ps1`

2. 把生成的 `offline-bundle\时间戳.zip` 发给目标机器

3. 目标机器解压后，双击 `Start-SketchShot.cmd`

4. 如需复查，双击 `Verify-SketchShot.cmd`

5. 如需卸载，双击 `Cleanup-SketchShot.cmd`

如果走绿色包路线，则对应为：

1. 源机器执行  
   `powershell -ExecutionPolicy Bypass -File .\scripts\Package-GreenBundle.ps1`

2. 把生成的 `green-bundle\时间戳.zip` 发给目标机器

3. 目标机器解压后，双击 `Run-SketchShot.cmd`

4. 如需停止，双击 `Stop-SketchShot.cmd`
