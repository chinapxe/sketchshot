## 1. 项目一句话说明

这是一个**以无限画布为载体的 AI 故事板创作工具**。  
用户围绕场次、角色、风格、九宫格动作和镜头组织故事板，并在镜头级触发图像或视频生成。

当前主线路已经明确：

- 真实能力以火山引擎 Ark 为主
- `Shot` 是故事板主执行节点
- `Continuity` 是九宫格连续动作规划节点，不是 9 个方案备选节点
- ComfyUI 不是当前优先事项
- Mock 适配器仅作为兜底和本地离线调试用途

## 2. 当前真实状态

### 2.1 故事板主链路已成型

当前前端已经具备以下故事板核心节点：

- `Scene` 场次节点
- `Character` 角色设定节点
- `Style` 风格设定节点
- `Continuity` 九宫格动作节点
- `Shot` 镜头节点

其中：

- `Shot` 用于把场次、角色、风格、参考图、首尾帧和九宫格动作统一整理为最终生成请求
- `Shot` 可生成图片或视频
- `Shot` 之间可以串接，表达镜头承接关系

#### 角色三视图链路补充

最近已经把 `ThreeViewGen -> Character` 的数据约定收拢为一条正式链路：

- 独立 `ThreeViewGen` 节点已可根据一张输入图生成角色三视图，支持输出“一张总览拼板图”或“三张独立图”
- 当 `ThreeViewGen` 输出总览图时，角色节点会同步得到 `threeViewSheetImage`
- 当 `ThreeViewGen` 输出独立图时，角色节点会同步得到 `generatedThreeViewImages`，并自动补到正式槽位 `threeViewImages`
- 角色节点 UI 现在明确区分“生成的三视图”和“人物三视图”，支持手动调整槽位，也支持一键“采用生成结果”
- `Shot`、`Continuity` 和资产中心当前都优先消费角色节点的正式三视图 `threeViewImages`，不再把临时生成结果当作最终口径

### 2.2 九宫格动作能力已经从字段升级为正式节点

当前九宫格动作节点不是简单文本框，而是完整节点能力，已经支持：

- 连接场次、角色、风格、参考图形成上下文
- AI 润色九宫格总提示词
- AI 自动拆解 9 格连续动作文字
- 生成一张九宫格预览图
- 将 9 格动作文本传给 `Shot`
- 将九宫格预览图传给 `ImageDisplay` 或 `VideoGen`

需要特别记住的业务约定：

- 连到 `Shot` 时，下游主要使用的是 9 格动作文字
- 连到 `ImageDisplay` 或 `VideoGen` 时，下游依赖的是九宫格节点生成出的预览图

### 2.3 模板系统已经进入“教学入口 + 用户复用”阶段

内置模板已经不是单纯示例数据，而是承担上手引导职责，当前已支持：

- 模板分类展示
- 推荐模板、适合类型、学习点、预设亮点展示
- 应用模板后的操作引导
- 用户把当前工作流保存为自己的模板
- 用户模板的加载、删除与再次复用

用户模板会自动清理：

- 上传资源
- 生成结果
- 运行状态
- 临时缓存

只保留结构、连线和节点设定，避免把旧结果一起带入新项目。

### 2.4 画布体验已补齐一批关键交互

最近这批已经实现的增强，后续维护时需要视为正式能力，而不是临时实验：

- 节点标题可直接重命名
- 左侧栏加宽，且暂时隐藏“常用预设”
- 从节点拉线到空白处松手，会弹出“当前允许连接的节点类型”菜单
- 连接句柄与连线按图片 / 视频 / 上下文等能力着色
- 连线动画仅在真实生成中生效，避免节点已完成但连线仍持续传动
- 帮助页用于解释节点关系、生图逻辑和生视频逻辑
- 关于页用于品牌与产品定位展示

### 2.5 本地项目持久化与导入导出已经可用

当前已支持：

- 浏览器本地草稿自动保存
- 页面刷新后自动恢复草稿
- 项目导出为带资源的文件包
- 项目重新导入并恢复节点、连线与资源引用
- 兼容历史 `wxhb` 与当前 `sketchshot` 项目格式

### 2.6 后端真实链路已经接通

后端当前已具备真实火山引擎链路：

### 2.7 交付链路已经具备正式形态

## 3. 系统如何跑起来

### 3.1 前端主干结构

前端运行主干是：

1. `App.tsx`
2. `Sidebar` 提供节点入口
3. `Canvas` 承载 React Flow 画布
4. `Toolbar` 承载工作流、模板、引擎配置、帮助等入口
5. `useFlowStore` 统一管理节点、连线和上下游同步

建议把 `useFlowStore` 理解成前端运行时的“状态总线”。

### 3.2 故事板镜头如何执行

以 `Shot` 节点为例：

1. 前端从 `useFlowStore` 取出当前节点和整张画布状态
2. `getShotContext` 收集上游 `Scene`、`Character`、`Style`、上游 `Shot`、`Continuity` 和参考图
3. `buildShotPrompt` 生成最终镜头 prompt
4. 前端先建立 `/ws/progress/{node_id}` 监听
5. 前端调用后端图片或视频生成接口
6. 后端创建异步任务
7. 任务进度通过 WebSocket 推送回前端
8. 节点状态、进度、输出结果和缓存签名被更新
9. 结果继续同步到下游节点

关键文件：

- `frontend/src/utils/storyboard.ts`
- `frontend/src/services/storyboardGeneration.ts`
- `frontend/src/stores/useFlowStore.ts`

#### 角色三视图链路如何流转

角色三视图相关的数据流建议按下面理解：

1. 参考图或上游图像先进入 `ThreeViewGen`
2. `ThreeViewGen` 按 `outputMode` 输出总览图或独立三张图
3. `useFlowStore` 在角色节点侧统一同步 `threeViewSheetImage`、`generatedThreeViewImages` 和 `threeViewImages`
4. 角色节点 UI 允许用户把生成结果作为正式三视图继续沿用，也允许手动改槽位
5. `Shot`、`Continuity`、资产中心与项目导出当前都以 `threeViewImages` 作为正式三视图入口

这条链路如果后续要改，优先一起检查 `ThreeViewGenNode`、`CharacterNode`、`useFlowStore.ts`、`storyboard.ts` 和 `assetCenter.ts`。

### 3.3 九宫格动作节点如何执行

九宫格动作节点有两条链路：

1. 文本规划链路  
   - 使用提示词服务生成总提示词  
   - 使用提示词服务拆成 9 格动作文字

2. 预览图生成链路  
   - 根据 9 格动作和上下文拼出预览图 prompt  
   - 调用图片生成接口  
   - 生成九宫格预览图并同步给下游

关键文件：

- `frontend/src/components/Nodes/ContinuityNode/index.tsx`
- `frontend/src/services/promptGeneration.ts`
- `frontend/src/services/continuityGeneration.ts`
- `frontend/src/utils/storyboard.ts`

### 3.4 连接关系如何校验

连接合法性不是散落在各个组件里，而是集中在：

- `frontend/src/utils/flowConnections.ts`

这里统一定义了：

- 每种节点允许连接到哪些下游节点
- `Shot` 输出为图片时能接什么
- `Shot` 输出为视频时能接什么
- `Continuity` 最多只能接入一个 `Shot`
- 句柄和连线应该显示什么颜色

后续如果要改节点关系，优先看这个文件。

### 3.5 后端任务如何执行

后端核心流程是：

1. API 接口接收生成请求
2. `TaskService` 创建任务并解析适配器
3. 适配器执行真实生成
4. 生成进度写入节点级 WebSocket 队列
5. 前端按节点 id 接收进度

关键文件：

- `backend/app/api/generate.py`
- `backend/app/api/ws.py`
- `backend/app/services/task_service.py`
- `backend/app/adapters/volcengine_adapter.py`

### 3.6 火山引擎配置如何生效

当前火山引擎配置有两层来源：

1. `backend/.env` 中的默认值
2. `backend/data/engine_config.json` 中的运行时覆盖值

用户通过前端工具栏保存引擎配置后，后端会：

- 写入 `engine_config.json`
- 重新构建并刷新运行中的 `volcengine` 适配器

所以现在日常使用不再依赖频繁修改 `.env`。

关键文件：

- `frontend/src/components/Toolbar/index.tsx`
- `backend/app/services/engine_config_service.py`

### 3.7 工作流、项目包和用户模板的区别

这是后续维护时很容易混淆的三类数据：

- 工作流  
  存在 `backend/data/workflows/`，用于常规保存 / 加载当前画布

- 项目包  
  通过 `projectExchange.ts` 导出导入，包含资源文件，适合迁移项目

- 用户模板  
  存在 `backend/data/templates/`，会清理资源和运行状态，只保留可复用结构

维护时不要把这三者混为一类。

## 4. 关键业务约定与易误解点

### 4.1 `Shot` 才是故事板主执行节点

虽然 `ImageGen` 和 `VideoGen` 仍然存在，但在当前产品定位下：

- `ImageGen` / `VideoGen` 更像通用能力节点
- `Shot` 才是正式故事板链路里的主执行节点

### 4.2 `Continuity` 不是 9 个候选方案

它表达的是：

- 一个镜头的 9 个连续动作节拍

而不是：

- 9 张互相竞争的方案图

### 4.3 `VideoGen` 不能凭空出视频

`VideoGen` 至少需要一张上游图片。  
如果用户只接了九宫格动作节点，但还没有先生成九宫格预览图，那么 `VideoGen` 仍然没有真正可用的图像输入。

### 4.4 `Shot` 的下游连接取决于 `outputType`

- `outputType=image` 时，`Shot` 可以接 `ImageDisplay` 或下一个 `Shot`
- `outputType=video` 时，`Shot` 可以接 `VideoDisplay` 或下一个 `Shot`

### 4.5 用户模板与项目导出不是一回事

- 用户模板是“结构复用”
- 项目导出是“项目迁移”

后续改数据清洗逻辑时，要分别确认两套行为是否仍符合预期。

### 4.6 文档可能落后于工作区代码

当前项目处在持续推进阶段。  
接手时如果发现：

- 文档和界面不一致
- 文档和某次提交说明不一致

优先看：

1. 当前工作区代码
2. `git status`
3. 最新的根目录文档

## 6. 本地运行与验证方式

### 6.1 后端

```powershell
Set-Location .\backend
pip install -r requirements.txt
python run.py
```

启动后检查：

- `http://localhost:8000/docs`
- `http://localhost:8000/api/health`

### 6.2 前端

```powershell
Set-Location .\frontend
npm install
npm run dev
```

访问：

- `http://localhost:3000`

### 6.3 关键说明

- 如果只修改了前端工具栏里的引擎配置，后端会即时刷新适配器，一般不需要改 `.env`
- 如果修改了 `backend/.env` 或 `backend/app/config.py`，则需要重启后端进程
- 前端不应硬编码 API 或 WebSocket 地址，统一由 `frontend/src/config/index.ts` 读取

### 6.4 代码级验证命令

后端：

```powershell
python -m unittest discover -s backend/tests -p "test_*.py"
```

前端：

```powershell
Set-Location .\frontend
npm test
npm run build
```

正式交付前，建议再按 `正式验收清单.md` 完整走一遍。
