## 1. 当前项目定位

SketchShot 当前已经明确为一个**以无限画布为载体的故事板创作工具**，而不是通用 AI 工作流编排平台。

当前统一理解为：

- 画布是“故事板工作区”，不是复杂流程引擎
- 节点优先表达故事内容、镜头关系、角色设定、风格约束
- AI 的职责是辅助创作和生成，而不是把用户带入底层参数编排
- 目标用户是“有想法、想快速搭建故事板的人”，使用体验更接近思维导图和分镜板，而不是 Dify、Coze、ComfyUI 这类平台

这一定义已经落到计划、节点设计、模板设计和后端接入策略上，后续不要再回摆到“做一个更复杂的流程平台”。

---

## 2. 到目前为止已完成的工作

## 2.1 故事板主语义与核心节点已落地

前端主干已经不再只是通用生成节点，故事板相关节点和关系已经进入可用状态：

- `Scene` 场次节点
- `Shot` 镜头节点
- `Character` 角色设定节点
- `Style` 风格设定节点

其中 `Shot` 已经成为当前最核心的执行节点，能够承接：

- 来自 `Scene` 的场次上下文
- 来自 `Character` 的角色设定与角色稳定性约束
- 来自 `Style` 的整体视觉风格约束
- 图像输出与视频输出两种生成模式

这意味着“故事板结构 -> 可执行镜头生成”的主链路已经建立。

## 2.2 视频相关关键节点能力已补齐到可用版本

围绕视频镜头创作，已经补齐了计划中最关键的一批能力：

- 首帧约束 `videoFirstFrame`
- 尾帧约束 `videoLastFrame`
- 镜头延续 / Shot to Shot 承接
- 连续镜头顺序表达
- 镜头版本比较基础能力

目前系统已经能围绕“连续片段创作”工作，而不再只是单次静态生成。

## 2.3 角色三视图与九宫格连续动作语义已纳入实现

与故事板创作直接相关的两个关键概念已经正式进入代码结构：

- `Character` 支持三视图图片结构 `threeViewImages`
- `Shot` 支持九宫格连续动作描述 `continuityFrames`

这里要特别记录一个业务定义，避免后续误解：

**九宫格在本项目中，不是 9 个方案备选，而是一个镜头的连续 9 个动作节拍，用于图生视频时描述动作推进。**

该语义已经落实在镜头 prompt 拼装和相关测试中。

## 2.4 模板体系已经从“示例”升级为“教学入口”

模板不再只是几套演示数据，而是开始承担“让用户快速理解这批新节点怎么串起来使用”的职责。

已经完成的方向包括：

- 模板元数据增强
- 模板分组展示
- 推荐模板与学习点提示
- 套用模板后的上手引导

这说明系统已经开始具备新用户教学能力，而不是只靠口头说明或文档介绍。

## 2.5 画布可用性与编辑体验已完成一轮增强

近期已推进的交互和体验增强包括：

- 节点折叠 / 展开能力，缓解长节点内容占用画布空间的问题
- 节点与连线混杂时的自动整理 / 散开能力，提升画布可读性
- 左侧节点选择区中文化，降低理解门槛

这些改动的目标不是“更炫”，而是让故事板画布在真实使用中更清晰、更像创作空间。

## 2.6 本地项目持久化、刷新恢复、导入导出已完成

围绕“用户不希望刷新就丢内容、希望项目可以保存与迁移”的需求，已经补齐：

- 浏览器本地草稿持久化
- 页面刷新后恢复当前画布内容
- 项目导入 / 导出
- 导出为可本地保存的文件包
- 导入后关联资源的恢复与重建

这部分已经不是临时缓存，而是进入了正式项目交付所需的基础能力层。

## 2.7 后端火山引擎主链路已真实接通并完成验证

后端现已完成火山引擎主适配器接入，并已打通以下真实能力：

- 提示词生成
- 文生图
- 图生图
- 图生视频

同时已经处理过一批关键工程问题：

- 默认适配器解析明确收敛到 `volcengine`
- 修正 `adapter=auto` 错误回退到 `mock` 的问题
- 图生视频超时策略已放宽到适合真实平台耗时
- WebSocket 任务进度推送链路可正常工作
- 节点执行完成后的按钮焦点残留问题已修复

当前后端不是占位实现，而是已经具备真实服务能力、可联调、可验证。

## 2.8 Docker 化与离线交付链路已完成编码收口

为满足正式交付和目标机器部署需求，已经完成一整套 Docker / 离线交付方案：

- `backend/Dockerfile`
- `frontend/Dockerfile`
- `docker-compose.yml`
- `docker-compose.offline.yml`
- `.env.docker.example`
- `.env.offline.example`
- `scripts/docker-build.ps1`
- `scripts/Package-OfflineBundle.ps1`
- `scripts/Deploy-OfflineBundle.ps1`
- `scripts/Verify-OfflineBundle.ps1`
- `scripts/Cleanup-OfflineBundle.ps1`
- 对应的 Linux shell 脚本
- 目标机器双击入口：
  - `Start-SketchShot.cmd`
  - `Verify-SketchShot.cmd`
  - `Cleanup-SketchShot.cmd`
  - `Build-OfflineBundle.cmd`

目前离线交付已经做到：

- 源机器可一键生成离线交付包
- 离线包内包含镜像 tar、compose、env、部署脚本、校验脚本、清理脚本
- 目标机器可通过一键入口完成导入镜像、启动容器、健康检查、输出访问地址

这一部分已经具备正式项目交付形态，而不是临时脚本拼接。

## 7. 如果下一步继续推进，建议顺序

建议按下面顺序恢复推进：

1. 先做一次正式验收
   - 在目标机器上验证离线包解压、部署、访问、火山引擎配置、结果落盘

2. 做一次 Git 提交
   - 将当前已经完成的故事板增强、持久化、导入导出、Docker 交付脚本统一落库

3. 继续做 M5 收尾
   - 完善模板教学体验
   - 完善交付文档
   - 补齐 README
   - 补齐异常态与空态说明

4. 再做一轮体验打磨
   - 继续提升画布复杂场景的可读性
   - 打磨资产中心 / 执行中心
   - 处理前端构建体积优化

不建议当前优先做的事情：

- 深度扩展 ComfyUI 对接
- 回到通用工作流平台方向
- 过早扩展过多技术型节点
